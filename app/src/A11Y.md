# Hyvemind Frontend Accessibility Conventions

This document covers the screen-reader conventions used across the React frontend. It is the single source of truth for *when to use `aria-live`*, *which politeness level*, and *what to do about loading / streaming / error states*. New components touching streaming output, async operations, or transient errors should follow these rules.

> **Audience**: anyone editing components under `app/src/components/` or `app/src/screens/`. Backend (Rust) accessibility is out of scope — this is about the Tauri webview UI only.

---

## TL;DR — When to use what

| Situation | ARIA pattern | Why |
|-----------|--------------|-----|
| LLM tokens streaming into a bubble | `aria-live="polite" aria-atomic="false"` on the bubble container, applied **only while streaming** | Polite queues behind user activity; non-atomic so each new chunk announces, not the whole bubble re-read on every token |
| Errors (modal, inline banner, toast) | `role="alert"` + `aria-live="assertive"` on the container | Assertive interrupts; `role="alert"` already implies live+assertive+atomic — combining the explicit attrs is belt-and-suspenders against older screen readers |
| Async button (Btn loading) | `aria-busy="true"` on the `<button>` while loading. No live region. | The button's *own* label is unchanged; we just signal "in flight". Live regions on buttons fight with focus context. |
| Long-running phase changes (Hivemind round, swarm phase, "Creating Plan…", "Decomposing Features…") | Visually-hidden `<span className="sr-only" aria-live="polite" aria-atomic="true">` containing a *single sentence* summary | Atomic so each new sentence reads cleanly; polite so it queues; sentence is debounced to phase changes, not per-token |
| Loading start (spinner appears) | Visible spinner with `role="status" aria-label="…"`. No live region. | Avoids announcing "loading…" on every render; SR users can navigate to the status role on demand |
| Loading completion (async work finishes) | A polite live region whose text flips from "" → "Done: <thing>" | The change is what announces — empty initial state means no spurious announcements on mount |
| Per-model rows updating in place (Hivemind live panel) | `aria-live="polite" aria-atomic="false"` on the *list container* | Non-atomic so SR only reads the row that changed; polite so it doesn't interrupt the user reading |

---

## The polite vs assertive decision

- **Polite (`aria-live="polite"`)** — default. SR waits for a natural pause. Use for *informational* updates: streamed tokens, status changes, completions, count changes.
- **Assertive (`aria-live="assertive"`)** — use **only** for errors and safety-critical interruptions. Use for: error modals, inline error banners, "your work was lost" warnings, retry-required notices.

If in doubt, **use polite**. Assertive interrupts the user and is jarring. Most SR users explicitly complain about over-use of assertive in modern web apps.

---

## The atomic vs non-atomic decision

- **`aria-atomic="true"`** — SR re-reads the *entire* region every time any descendant changes. Use when:
  - The region holds a short, self-contained sentence (e.g. "Round 2 of 3 — 4 of 4 models complete")
  - You debounce updates so re-reading the whole thing is cheap
  - The region is `role="alert"` (which sets atomic=true automatically)
- **`aria-atomic="false"`** (default) — SR reads only the changed subtree. Use when:
  - The region is large (a chat bubble, a list of rows)
  - Updates are frequent (token streaming, per-model status flips)
  - Re-reading the whole region every update would flood the user

For LLM streaming: **always non-atomic.** Re-reading a growing bubble on every token is unusable.

## What is implemented today (Fix 9.1)

| File | Region | Pattern |
|------|--------|---------|
| `components/ErrorModal.tsx` | Modal content | `role="alert"` + `aria-live="assertive"` + `aria-atomic="true"` |
| `components/ActivityStream.tsx` | Streaming assistant chat bubble | `aria-live="polite" aria-atomic="false"` (only while `isStreamingNow && !me`) |
| `components/ActivityStream.tsx` | `ErrorRow` (inline conversation errors) | `role="alert" aria-live="assertive"` |
| `components/ActivityStream.tsx` | Delimiter loading banner ("Creating Plan…", etc.) | `aria-live="polite" aria-atomic="true"`; inner spinner gets `role="status" aria-label="Loading"` |
| `components/ActivityStream.tsx` | Bare streaming spinner | `role="status" aria-label="Streaming response"` (no live region — the bubble above is authoritative) |
| `components/HivemindReviewLivePanel.tsx` | Visually-hidden phase/round summary | `<span className="sr-only" aria-live="polite" aria-atomic="true">` with a sentence-level summary that changes only on phase transitions |
| `components/HivemindReviewLivePanel.tsx` | `between_rounds` transition announcement | Reads "Round {N} merged, preparing next round" the instant the backend emits `merge_completed` for round N, then yields to either the next `round_started` summary or the final `completed` summary. Polite + atomic so the short-lived phase is read cleanly without interrupting the streaming announcement that follows. |
| `components/HivemindReviewLivePanel.tsx` | Per-model rows container | `aria-live="polite" aria-atomic="false"` |
| `components/HivemindReviewLivePanel.tsx` | Per-failure error detail | `role="region"` + `aria-label` referencing reviewer label; `aria-live="off"` so the full error is navigable but not auto-read. The existing `sr-only` summary announcement remains the authoritative live notification. |
| `screens/Tasks.tsx` | Transient task error banner | `role="alert" aria-live="assertive"` |
| `screens/Tasks.tsx` | Retry-failed review error text | `role="alert" aria-live="assertive"` |
| `components/atoms.tsx` `Btn` | Loading state | `aria-busy="true"` on the `<button>` when `loading` prop is true. No live region. |

## Anti-patterns (don't do these)

- **Don't** put `aria-live` on a spinner. Use `role="status" aria-label="…"` so SR users can opt in by navigating to it.
- **Don't** announce "Loading…" on every render. Toggle the announcement only on real transitions (mount → loaded, or empty → done).
- **Don't** put `aria-atomic="true"` on a streaming chat bubble. SR will re-read 800 tokens every token.
- **Don't** mix `role="alert"` with `aria-live="polite"`. They contradict — `role="alert"` implies assertive.
- **Don't** announce success on every save. Reserve assertive for *user-actionable* events (errors, required confirmations).
- **Don't** add `aria-live` to a region that already contains focus. Focus context is what SR is reading; an overlapping live region creates double-speak.

## Testing checklist (manual)

We don't have automated SR snapshot tests. When you touch a region that uses any of the patterns above, manually verify with **macOS VoiceOver**:

1. **Enable VoiceOver**: `Cmd + F5` (or `Touch ID` triple-press on supported Macs).
2. **Open the app** (`npm run tauri dev`).
3. **Stream a Task**:
   - Send a prompt → confirm the streaming bubble announces incremental chunks (not the whole bubble re-read).
   - Trigger an error (e.g. disconnect network mid-stream) → confirm the inline error interrupts.
4. **Trigger an ErrorModal**:
   - Throw a Hyvemind error (e.g. invalid swarm config) → confirm VO interrupts and reads the title + detail.
5. **Run a Hivemind review**:
   - Start review on a plan → confirm phase transitions ("Gathering context", "Round 1 of 2: 0 of 4 models complete", "Round 1 of 2: 4 of 4 models complete", "Merging round 1 feedback", "Hivemind review complete") all read politely.
   - Confirm per-model row updates (status flips, token counts) do **not** flood — only the changed row should announce.
6. **Click a loading button**:
   - Click any `Btn` that goes into loading state → confirm VO reads "busy" attribute change but does not re-announce the button label.

If a region misbehaves, prefer fixing the *region semantics* (live politeness, atomic, debouncing the text) over silencing it with `aria-hidden`.
